Proposed PyPI Library Plan

Recommended PyPI library name

kvfleet

Why this name
	•	kv signals your strongest differentiator: KV-cache-aware routing for in-house hosted LLMs.
	•	fleet signals orchestration across a fleet of self-hosted models, not just one endpoint.
	•	It is short, memorable, technical, and clearly positioned for infrastructure teams.
	•	It also avoids sounding like a generic “prompt router” and instead sounds like a serious production control plane.

Backup name options
	•	fleetroute
	•	aegisroute
	•	privroute
	•	cachepilot
	•	modelmux

Naming check

I did not find an obvious existing PyPI package or well-known Python library dominating under kvfleet, while related spaces are occupied by packages like litellm, semantic-router, routellm, semroute, routeagent, and llm-router. That said, package-name availability can change at any time, so you should do one final PyPI publish-time check before locking the name.  ￼

⸻

One-line positioning

kvfleet is a production-grade, KV-cache-aware, hardware-aware intelligent routing library for self-hosted and hybrid LLM fleets, designed to route each request to the best in-house model based on quality, latency, cost, policy, and live cluster state.

⸻

Why this library should exist

AWS Bedrock Intelligent Prompt Routing is useful, but it is built for routing across models within the same model family and AWS positions it around quality/cost optimization inside Bedrock. AWS also documents important constraints, including English optimization and routing behavior that is not designed around your private serving telemetry or application-specific feedback loops.  ￼

Existing Python tools each cover only part of the problem:
	•	LiteLLM is strong for unified provider access, retries, fallbacks, and load balancing, but its routing is primarily provider/deployment orchestration rather than deep KV-cache and GPU-state-aware in-house serving control.  ￼
	•	semantic-router / semroute / llm-router focus on semantic dispatch using embeddings, which is useful, but that is not enough for enterprise in-house model routing under live capacity pressure.  ￼
	•	RouteLLM is closer to learned cost/quality routing, but it explicitly focuses on routing between a stronger expensive model and a cheaper weaker model, which is narrower than your in-house fleet use case.  ￼

That creates a strong opening for a library focused on:
	•	self-hosted models
	•	KV-cache affinity
	•	hardware-aware routing
	•	policy-aware routing
	•	continuous shadow evaluation
	•	learned routing from your own traffic

⸻

Scope of the library

Core scope

A Python library and optional OpenAI-compatible gateway that sits in front of:
	•	self-hosted vLLM clusters
	•	NVIDIA Triton inference endpoints
	•	TGI-style endpoints
	•	Ollama or internal model gateways
	•	optional Bedrock or external fallback models

It should intelligently decide:
	1.	which model to use,
	2.	which replica to send traffic to,
	3.	when to escalate from a cheaper/faster model to a stronger model,
	4.	when to keep traffic private, and
	5.	how to optimize around live cluster conditions.

Explicit non-goals for v1
	•	not a training framework
	•	not a full observability platform
	•	not a general-purpose LLM app framework
	•	not just another OpenAI wrapper

Product vision

Become the de facto routing control plane for enterprise in-house LLM deployments.

The project should be known for three things:

1. KV-cache-aware routing

Not just “pick a model,” but “pick the exact worker that already has the relevant prompt/session cache.”

2. Hardware-aware routing

Use live cluster state:
	•	queue depth
	•	GPU memory pressure
	•	active sessions
	•	token throughput
	•	time-to-first-token estimates
	•	pending batch size
	•	model warmness / cold-start status

3. Outcome-aware routing

Continuously learn from:
	•	offline evals
	•	model comparisons
	•	shadow traffic
	•	user feedback
	•	escalation rates
	•	task success metrics

Feature roadmap to make it production-grade

Phase 1 — MVP

1. Unified model registry

A registry for all routeable models with metadata such as:
	•	model name
	•	endpoint URL
	•	provider type
	•	context window
	•	tokenizer metadata
	•	estimated cost
	•	latency priors
	•	quality priors
	•	allowed data classes
	•	supported tools / JSON mode / streaming

2. Routing strategies

Built-in routing modes:
	•	static route
	•	weighted route
	•	rules-based route
	•	semantic route
	•	cost-first route
	•	latency-first route
	•	cheap-first cascade
	•	quality-first route
	•	hybrid score route

3. In-house model support

Adapters for:
	•	vLLM
	•	Triton
	•	TGI
	•	Ollama
	•	generic OpenAI-compatible endpoints
	•	generic HTTP adapter for proprietary internal inference services

4. KV-cache affinity routing

This should be your flagship feature.
	•	session-aware routing
	•	prefix-aware routing
	•	consistent hashing by conversation/system-prompt fingerprint
	•	cache locality scoring
	•	route-to-replica if likely warm cache exists
	•	fallback when target worker is unhealthy

5. Telemetry-driven routing inputs

Read runtime signals such as:
	•	queue depth
	•	average TTFT
	•	tokens/sec
	•	batch size
	•	active requests
	•	memory usage
	•	GPU VRAM headroom
	•	health check score

6. Fallback and retry policy
	•	retry on transient failures
	•	promote to stronger model on timeout
	•	structured fallbacks by route class
	•	degrade gracefully to safe default model

7. Full routing trace

Every route decision should explain:
	•	why model X was selected
	•	why model Y was rejected
	•	what signals were considered
	•	what policy gates were applied
	•	whether cache affinity influenced the decision

This is essential for enterprise trust.

8. Config-first developer experience
	•	Python API
	•	YAML config
	•	env var overrides
	•	policy bundles
	•	route templates for common setups

⸻

Phase 2 — Strong differentiators

9. Multi-objective scoring engine

A weighted routing function that combines:
	•	expected quality
	•	expected latency
	•	expected cost
	•	compliance risk
	•	cache affinity score
	•	hardware saturation score

This should be customizable per team, tenant, or workload.

10. Task and domain classification

Pre-route classification for:
	•	summarization
	•	Q&A
	•	extraction
	•	coding
	•	SQL generation
	•	legal/regulated response
	•	agent planning
	•	multilingual prompts

This lets you send coding prompts to code-specialized models and regulated prompts to approved private models only.

11. Policy-aware routing

Hard enterprise guardrails:
	•	PII-bearing traffic must remain on private models
	•	certain teams can use only approved models
	•	prompts tagged “regulated” require specific route classes
	•	confidential prompts must not leave VPC
	•	data residency rules by geography/business unit

12. Prompt fingerprinting
	•	normalize prompt structure
	•	hash system prompt + retrieved context signature
	•	improve cache locality
	•	enable route memory and similar-request treatment

13. Semantic similarity router

A fast CPU-side semantic router for:
	•	model family selection
	•	domain specialist selection
	•	route class selection

Useful, but not your core differentiator.

14. Request budgeting and quotas
	•	per team budget caps
	•	per route budget caps
	•	spend-aware routing
	•	“stay under monthly budget” routing mode
	•	emergency cost-throttle mode

⸻

Phase 3 — Enterprise production features

15. Shadow traffic and canary rollouts

This is a must-have.
	•	mirror a percentage of traffic to new candidate models
	•	keep user-facing answer on stable model
	•	compare candidate outputs offline
	•	score drift across latency, quality, and structure
	•	promote/demote models based on evidence

16. Offline replay engine

Replay production prompt logs against alternate models and routing policies:
	•	compare cost
	•	compare latency
	•	compare task success
	•	compare route choices
	•	estimate rollout impact before deployment

17. Continuous route calibration
	•	learn from observed outcomes
	•	adjust thresholds
	•	tune route weights
	•	update quality priors by domain
	•	detect route drift after new model versions

18. Safety and compliance pipeline
	•	PII detection
	•	redaction / pseudonymization
	•	prompt transformation before external route
	•	response rehydration
	•	audit logs for policy decisions

19. SLO-aware routing

Respect operational objectives:
	•	target p95 latency
	•	target max cost/request
	•	target minimum quality score
	•	emergency load shedding
	•	degrade to fast model under overload

20. Tenant-aware routing

Different business units can define:
	•	preferred models
	•	blocked models
	•	latency priorities
	•	compliance constraints
	•	spending limits

⸻

Phase 4 — Advanced moat features

21. Learned intelligent router

Train a lightweight router from your own enterprise traffic using:
	•	prompt embeddings
	•	prompt length
	•	route class
	•	prior model performance
	•	feedback labels
	•	successful completion signals
	•	escalation outcomes

This is your answer to “Bedrock-like routing, but for our own models.”

22. Reinforcement or bandit-based routing

Instead of static rules, use:
	•	contextual bandits
	•	Bayesian optimization
	•	online threshold tuning
	•	reward functions based on business outcome

This is v2/v3 material, not MVP.

23. Generative semantic cache

For near-duplicate prompts:
	•	detect similar prior work
	•	retrieve candidate responses
	•	optionally synthesize with a small fast local model
	•	bypass heavy model when safe

24. Route verification hooks

Optional post-generation checks:
	•	JSON schema validation
	•	hallucination heuristics
	•	toxicity filters
	•	policy validators
	•	execution validators for code/SQL/tool outputs

25. Auto-escalation chains

If output confidence is low:
	•	escalate from small model to medium model
	•	then strong model
	•	then verifier model
	•	then human-review queue

What makes kvfleet “top in the game”

To be top-tier, it should not market itself as “smart prompt router.” That is too generic.

It should own this category:

KV-cache-aware control plane for enterprise LLM fleets

That gives you a sharper and more defensible identity than LiteLLM or semantic-router, because those tools are broader or different in scope. LiteLLM emphasizes unified provider access and routing/load-balancing. semantic-router emphasizes semantic decision making. RouteLLM emphasizes cost-quality routing between two models. Your space is in-house serving intelligence.  ￼

Your strongest moat features are:
	1.	KV-cache affinity routing
	2.	hardware-aware routing
	3.	policy-aware routing
	4.	shadow rollout + replay evaluation
	5.	learned routing from enterprise traffic

That combination is the winning story.
